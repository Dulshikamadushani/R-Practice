### renv package

In R, packages are normally installed in a global library, meaning the
same packages are shared across all projects. Because package versions
may differ between computers, this can cause code to break when a
project is moved to another machine.

This creates problems for reproducibility, since the same code may not
produce the same results if package versions are different.

The renv package solves this problem by creating a project-specific
library that records the exact versions of packages used in the project.

This ensures that the project environment can be recreated on other
computers, improving reproducibility and portability of the code.

### Steps followed:

#### Create a GitHub repository

First, a GitHub repository was created to store the project files. For
this demonstration, I used the already created class repository.

#### Load the necessary packages

The required packages were loaded in R to demonstrate how renv manages
package versions.

``` r
library(ggplot2)
library(dplyr)
```

#### Initialize the renv project environment

A project-specific library and lock file were created using renv.

- Run the code in the console: renv::init()
- Record the package versions used in the project: renv::snapshot()
- These commands create a project-specific library and a lock file
  ‘(renv.lock)’ that records the exact versions of all packages used.

#### Commit the renv files to GitHub

The renv.lock file and related project files were committed and pushed
to GitHub so that the project environment can be reproduced elsewhere.

#### Recreate the environment on another computer

To run the project on a different computer:

- Download the repository from GitHub.
- Open the project in RStudio.
- Install renv if it is not already installed with:
  install.packages(“renv”)
- Restore the project environment using: renv::restore()

This installs the exact package versions listed in the renv.lock file,
ensuring the project runs the same way on different machines.
